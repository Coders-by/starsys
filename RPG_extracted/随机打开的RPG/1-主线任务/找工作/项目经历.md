#### **功能安全架构设计与落地**

**核心目标**：构建符合ISO 26262 ASIL-B要求的故障监控与降级体系，保障车辆在异常工况下的可控性。

---

#### **1. 需求分析与初期实现**

- **问题场景**：需实时监控EPS扭矩异常、制动斜率超限等20+故障信号，触发安全降级策略（如退出自动驾驶、紧急制动）。
    
- **初期方案**：
    
    - **直接实现**：在代码中硬编码故障判断逻辑，直接调用事件上报接口。
        
    - **痛点**：逻辑分散在10+业务模块，新增需求需跨文件修改，代码耦合度高，维护成本增加50%。
        
- **个人贡献**：快速完成初期需求交付，但主动识别架构缺陷，推动团队重构方案。
    

---

#### **2. 架构升级：观察者模式与模块化设计**

- **重构方案**：
    
    - **设计模式**：引入`FaultDetector`抽象接口，定义统一的故障检测流程（`detect()`→`reportEvent()`→`applyDegradation()`）。
        
    - **模块解耦**：实现`CarStateFaultDetector`（车速/挡位异常）、`ControlFaultDetector`（转向/制动指令超限）等6类检测器，通过订阅-通知机制与主流程解耦。
        
    - **扩展性提升**：新增故障类型仅需实现新Detector，代码修改范围减少70%。
        
- **技术细节**：
    class FaultDetector {
    public:
        virtual void detect(const VehicleState& state) = 0;
        virtual void attachHandler(EventReporter& reporter) = 0;
    };
    class BrakeSlopeDetector : public FaultDetector {
        void detect(const VehicleState& state) override {
            if (state.brake_slope > threshold_) {
                reporter_.publish(SafetyEvent::BRAKE_OVERRUN);
            }
        }
    };
    
- **成果**：
    
    - 代码复用率提升至80%，故障检测响应延迟<10ms；
        
    - 支持动态加载检测策略，适配长城、比亚迪等多车型需求。
        

---

#### **3. 系统级安全架构优化（车规级方案）**

- **问题发现**：
    
    - 在实车路测中，发现**SOC端Linux系统软实时性不足**导致的降级失效（如SOME/IP通信卡顿引发油门接管延迟）。
        
    - **根因分析**：SOC端资源竞争导致关键指令（如退出自动驾驶）未能及时下发至MCU。
        
- **解决方案**：
    
    - **架构迁移**：主导将核心降级逻辑（如驾驶员接管检测、紧急制动）移植至MCU端，利用其硬实时特性（μC/OS-III RTOS）保障响应确定性。
        
    - **双冗余设计**：
        
        - SOC端：轻量化监控，仅做预警；
            
        - MCU端：实现ASIL-D级安全逻辑，独立于SOC运行，故障响应延迟<5ms。
            
- **落地效果**：
    
    - 解决“踩油门无响应”等5类量产级问题，通过OEM功能安全评审；
        
    - 支持毫秒级故障接管，满足ISO 26262硬件随机失效度量（PMHF<1e-8）。
